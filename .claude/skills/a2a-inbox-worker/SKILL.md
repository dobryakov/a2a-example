# A2A: агент-исполнитель (одна задача за вызов)

Этот скилл задаёт **одну атомарную активацию** исполнителя поверх **`docs/SYSTEM_INSTRUCTIONS.md`** и **`docs/ATOMIC_CAPTURE.md`**. Шаги **B, C, D** и формат выдачи в `outbox/` — **буквально** по нормативу (включая **§2.3 «Паттерн Конверта»** и **раздел 7** для запросов с **`parent_id`**).

**Инвариант скилла:** за **один** вызов разрешено успешно завершить **не более одной** задачи (успешный **D**). После успешного **D** вызов **немедленно** завершается. **Poll**, sleep между проходами и «докрутка» всей очереди в цикле — **не** входят в скилл; их делает **внешняя** среда (cron, systemd, скрипт, человек), снова активируя воркер.

---

## 0. Нормативная иерархия

1. При расхождении побеждают **`docs/SYSTEM_INSTRUCTIONS.md`** и **`docs/ATOMIC_CAPTURE.md`**.
2. Форматы запроса/ответа, **итерации (`parent_id`)** и публикация в `outbox/` — **`SYSTEM_INSTRUCTIONS` §2–3 и §7**.

---

## 1. Предусловия (как в `SYSTEM_INSTRUCTIONS` §1)

Перед работой в **этом** вызове:

1. Определи **`$REPO`** — корень Git-репозитория A2A.
2. Прочитай **`config/a2a-environment.json`**, пути каталогов относительно `$REPO`.
3. Загрузи манифест **`.well-known/agents/<agent_id>.json`** (по умолчанию **`product_manager.json`**). Извлеки **`agent_id`**, **`capabilities`**, **`inbox_path`**; `inbox_path` согласован с конфигом.
4. Не трогай чужие файлы в `processing/`, кроме цепочки своего успешного захвата.

---

## 2. Сортировка кандидатов (фиксирована для этого скилла)

Имена `inbox/*.json` сортируются **лексикографически по полному имени файла** (как одна из опций **§3 шаг A.1** норматива); стратегия не меняется между вызовами данного профиля воркера.

---

## 3. Самоотбор до захвата

Прочитай кандидат из `inbox/` **без** `rename`, пока не решено «принимаю».

**Итерация (`parent_id` в JSON кандидата):** до решения «принимаю» проверь:

- Поле **`id`** кандидата **не равно** **`parent_id`** (иначе постановка противоречит разделу 7 — эскалация человеку, **не** захватывай).
- В `$REPO/outbox/` существует **хотя бы один** файл **`res_{parent_id}.*`** (машинный ответ предка по нормативу). Если **нет** — родительская итерация **не опубликована**; **не захватывай** этот файл, зафиксируй путь проверки и эскалируй человеку (кандидат остаётся в `inbox/`). **Не** подменяй базовый контекст угадыванием из `archive/` без **`res_{parent_id}.*`**.

В остальном:

- **Не принимаю** — следующий файл в порядке §2 (**без** изменения `inbox/`).
- **Принимаю** — **одно** **`rename`** `inbox/{id}.json` → `processing/{id}.json`** (**`docs/ATOMIC_CAPTURE.md`**).

При `ENOENT`/гонке — следующий кандидат того же списка. При `processing/{id}.json` уже существует — эскалация человеку.

---

## 4. Одна активация: шаги

### 4.1. Git перед работой

**`git pull`** (или политика, заданная пользователем). Если pull нельзя выполнить из‑за **отсутствия `remote`** или **нет upstream** у текущей ветки (и аналогичные сообщения Git без реального слияния) — **не** считай это провалом прохода и **продолжай** с шага A. При **конфликте содержимого** в `inbox/` / `processing/` / `outbox/` после pull или слияния — **стоп**, человек (**`SYSTEM_INSTRUCTIONS` §5**).

### 4.2. Шаг A и ровно один пайплайн

Пока **успешный D для какого‑либо `request_id`** в этом вызове **ещё не** выполнен:

1. Перебирай **`inbox/*.json`** по §2 (в одном проходе норматива см. ограничения **§3 инвариант**).
2. Для каждого: валидация **§2.1**, самоотбор §3 при невалидности — человек (**не** захватывать молча проблемное как своё без политики).
3. После первого **успешного** захвата для принятой задачи перейди к **B**, затем **C**, затем **D** по нормативу.
4. После успешного **D** — **немедленно заверши выполнение скилла** (никаких дальнейших кандидатов, второго commit за этот вызов).

Если список исчерпан **без** успешного **D** — заверши вызов («пустой тик»); внешний оркестратор решит следующий запуск.

### 4.3. Шаги B и C (`SYSTEM_INSTRUCTIONS` §3)

- **B (в т.ч. `parent_id`):** чтение **`processing/{request_id}.json`**. Если в запросе есть **`parent_id`**:
  1. **Базовый контекст** уже частично проверен до захвата (раздел 3); после захвата прочитай **`outbox/res_{parent_id}.*`** целиком в рабочую память шага B.
  2. Если есть **`outbox/{parent_id}/`**: прочитай **`metadata.json`**, затем **каждый** вложенный файл из массива **`files`** (как в **§2.3.5**). Это часть базового контекста, а не редактируемая черновая копия.
  3. Прочитай **`archive/{parent_id}.json`**, если нужно согласовать исходную постановку предка с полями **`params`** текущего запроса (дельта в **`params.comment`** / **`change_request`** и т.д.).
  4. **Склейка для исполнения:** трактуй базовый контекст как **несбрасываемое уже сделанное**; применяй **`method`/`params` текущего запроса** как поверх лежащее уточнение или исправление (подробнее **§7.3**). В **`result`** нового ответа при существенных отличиях кратко укажи, что унаследовано от предка и что изменено итерацией (audit trail одним коммитом `Processed {request_id}`).
  5. **Запрет:** не изменять **`outbox/res_{parent_id}.*`**, **`outbox/{parent_id}/`**, **`archive/{parent_id}.json`**; новая выдача только **`res_{request_id}.*`** и при необходимости **`outbox/{request_id}/`**.
- Затем (как в любом запросе): исполнение `method`/`params` в рамках **`capabilities`**, объект ответа **§2.2**.
- **Перед C:** по **§2.3** — **всегда** планируй **`outbox/res_{request_id}.json`** (или иной `ext`). Если есть **дополнительные артефакты**, добавь конверт **`outbox/{request_id}/`** с **`metadata.json`** и вложениями; собирай его в **`$REPO/.a2a-outbox-staging/{request_id}/`**, затем атомарный **`rename`** в `outbox/{request_id}/`. Дубликат JSON-RPC внутри конверта (**`response.json`**) **не** создавай.
- **C:** опубликовать **`res_{request_id}.{ext}`**, при необходимости — конверт; затем **`processing/{request_id}.json` → `archive/{request_id}.json`** только когда готово всё по **§3 шаг C** норматива.

**Целостность C (конверт и `res_`):**

- **До** переноса в `archive/` убедись, что **`outbox/res_{request_id}.{ext}`** существует, не пустой и парсится как JSON с объектом **§2.2**; при конверте — что **`outbox/{request_id}/metadata.json`** на месте и **каждый** файл из его **`files`** физически присутствует в этом каталоге.
- Если при публикации произошла ошибка **после** частичной записи в `outbox/` (например, конверт уже переименован, а запись **`res_*`** оборвалась): **не** переносить запрос в `archive/`; **не** считать шаг D выполненным; убрать или дозаполнить артефакты в том же проходе либо эскалировать человеку с путями и состоянием дерева.
- Короткие скрипты «всё одним блоком» допустимы, но **разделяй** фазы: сначала доводи до конца стейджинг конверта и/или `res_*.tmp` → `res_*`, затем проверяй целостность; так снижается риск сценария «конверт есть, а `res_` нет».

Не клади машинный ответ в корень `outbox/` под именем `{id}.json` без префикса `res_` (запрет **§2.3.1**).

### 4.4. Шаг D

**`git add`** по изменённым путям задачи (**§3 D**), **`git commit`** с сообщением, равным литералам `Processed ` и строковому `id` запроса (пример полного сообщения для `abc-123`: `Processed abc-123`), затем **`git push`**, если настроен push destination (**`remote`** и политика позволяют); если push невозможен (например, remote не задан) — шаг D после **успешного commit** считается выполненным для этого прохода. При non-FF без force — см. норматив **§5**.

---

## 5. Запреты (см. `SYSTEM_INSTRUCTIONS` §6)

- Не `rename` до решения «принимаю».
- Не публиковать в `outbox/` до успешного захвата.
- Не обрабатывать **вторую** задачу до конца вызова после успешного **D**.
- Не перезаписывать готовую выдачу **`outbox/res_{id}.*`** или каталог **`outbox/{id}/`** без оператора (**в т.ч.** выдачу **`parent_id`** при итерации, §7).
- Не оставлять в корне **`outbox/`** «брошенные» **`res_*.tmp`** или заведомо битые пустые файлы после сбоя: удали временный файл или доведи атомарную **`rename`** до **`res_{id}.*`**; иначе эскалация.
