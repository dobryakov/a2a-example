# A2A: задача из произвольного текста → `inbox/`

Заготовка скилла для **Claude Code / совместимых** окружений. Истина формата — в `docs/SYSTEM_INSTRUCTIONS.md` (поля запроса).

## Граница роли (обязательно)

Этот скилл — **только постановщик задачи** (оркестраторский шаг «положить запрос в очередь»). Он **не** является исполнителем очереди.

**Делать:** записать **ровно один** JSON-файл запроса в `inbox/{id}.json` (или черновик в `docs/drafts/`, если пользователь просил не трогать рабочую очередь). В `params` можно явно описать желаемый формат выдачи (в том числе требование **конверта** `outbox/{id}/`, перечень артефактов, ссылки на разделы `docs/SYSTEM_INSTRUCTIONS.md`) — это **текст постановки для будущего воркера**, а не команда выполнить работу в этом же вызове.

**Не делать в рамках этого скилла:** атомарный захват (`inbox/` → `processing/`), работа по задаче, создание **`outbox/res_{id}.*`**, каталога-конверта **`outbox/{id}/`**, перенос запроса в **`archive/`**, коммиты/пуш от имени завершения задачи. Всё это относится к **воркеру**, см. `.claude/skills/a2a-inbox-worker/SKILL.md`. Не «добивать удобство» за пользователя, выполняя задачу сразу после постановки.

## Когда активировать скилл

Пользователь дал неструктурированную постановку («сделай задачу…», «запись в очередь…», сырое ТЗ в чат, **доработай задачу X…**) и явно или по контексту хочет **файл-задачу** для этого репозитория A2A.

## Доработка существующей задачи: как распознать и оформить

**Смысл:** воркер по нормативу **обязан** знать **`parent_id`** (см. `docs/SYSTEM_INSTRUCTIONS.md` раздел 7). Постановщик **вытаскивает** из текста пользователя факт итерации и **кладёт в JSON** верные **`id`** и **`parent_id`** — без выполнения работы воркера.

### Сигналы, что речь о доработке (итерации)

Интерпретируй как итерацию, если пользователь (включая формулировки на русском и английском) **плюс к новой формулировке работы** намекает на опору на уже сделанное, например:

- «доработай / поправь / уточни / дополни / пересобери **по прошлой задаче**»;
- «**повтор / вторая итерация**, **feedback**, **revision**»;
- ссылка на **конкретный идентификатор**: UUID, `req-…`, имя файла `…/outbox/res_<id>.json`, путь `outbox/<id>/`, «задача `<id>`»;
- «**на базе** ответа / конверта / того, что уже выложили в outbox»;
- **продолжи** / **same task** / **follow-up** с указанием или явным контекстом предыдущего `id` из чата или репозитория.

Если сигнал есть, но **какой** именно предок — неочевиден, **спроси пользователя одним коротким вопросом**: «Укажи `id` предыдущей итерации (как в имени `archive/{id}.json` или `outbox/res_{id}.json`).» Не угадывай произвольный UUID.

### Как заполнить поля при итерации

1. **`parent_id`** (строка): **ровно** тот **`id`**, на чью **опубликованную** выдачу опираются (узел-родитель в дереве итераций).
2. **`id`**: **новый** уникальный идентификатор; **никогда** не копируй `parent_id` в `id`. Для итерации по умолчанию — **новый UUID v4** (имя файла `inbox/{id}.json` совпадает с этим `id`). **Не** добавляй суффиксы вроде `-v2` в `id`: реляция задаётся только **`parent_id`** (пример: `docs/examples/request-iteration-with-parent-id.json`).
3. В **`params`** обязательно отрази дельту отдельно от «полного промпта»:
   - человекочитаемо: **`comment`**, **`change_request`**, **`delta`**, **`acceptance_notes`** (выбери подходящие ключи по смыслу);
   - дублируй суть в **`raw_prompt`** / **`freeform`**, если остальной текст длинный, чтобы воркеру было что читать целиком.
4. **Лёгкая самопроверка по репозиторию (только чтение, без воркера):** убедись, что существует **`outbox/res_{parent_id}.json`** (или иной **`outbox/res_{parent_id}.*`** из норматива). Если файла нет — **не ставь** задачу как готовую итерацию молча: сообщи пользователю, что предок не опубликован или не найден, и что нужен корректный `parent_id` или дождаться завершения родительской задачи.

**Запрещено в этом скилле:** править **`outbox/`**, **`archive/`**, **`processing/`**, чтобы «подготовить» предка — только постановка нового запроса в **`inbox/`**.

## Обязательные шаги (без упрощения протокола)

1. Прочитай `docs/SYSTEM_INSTRUCTIONS.md` раздел про **формат запроса** (§2.1) и **Iteration Policy** (раздел 7): `jsonrpc`, `method`, `params`, `id`, опционально **`parent_id`**, `sender_id`, `timestamp`.
2. Определи, **новая ли это задача с корня** или **итерация** (раздел **«Доработка существующей задачи»** выше). Для **корня**: сгенерируй новый **`id`** (по умолчанию **UUID v4**); поле **`parent_id` не включай**. Для **итерации**: задай **`parent_id`** и **новый** **`id`** (см. тот же раздел).
3. Имя файла **обязано** быть ровно `inbox/{id}.json` (значение `id` без фигурных скобок).
4. Собери тело файла как один JSON-объект без лишнего текста до/после.
5. Значения:
   - **`jsonrpc`**: `"2.0"`.
   - **`method`**: выбери по смыслу текста пользователя или спроси; если метод не очевиден — используй `task.enqueue` и положи интерпретацию в `params`; для продуктовых формулировок допускаются префиксы вроде `product.*`. Для доработки часто уместны методы с семантикой **refine** / **revise** / повтор исходного `method` родителя — если из контекста ясно, что воркер должен продолжить тот же тип работы, можно отразить это в **`params.expected_method_hint`** (неформальное поле для человека/воркера; норматив его не требует).
   - **`params`**: **объект**. Для **новой** задачи: минимум **`raw_prompt`** или **`freeform`**. Для **итерации**: плюс явная **дельта** (например **`comment`**, **`change_request`**), см. раздел выше.
   - **`sender_id`**: по умолчанию `human` или из явного указания пользователя.
   - **`timestamp`**: текущее время в **UTC** в форме ISO-8601 (например `2026-05-20T15:04:05.123Z`).
6. **Запись файла:**
   - по умолчанию — **`inbox/{id}.json`** в корне репозитория;
   - если пользователь попросил **только черновик** — положи в **`docs/drafts/inbox-{id}.json`** и явно сообщи путь (**не** в рабочей очереди `inbox/`).
7. Если пользователь просил зафиксировать в Git — после записи выполни только то, что он просил (например `git add` выбранного файла и `git commit`). Не добавлять секрет в репозиторий.

### Ответ в чат после постановки (обязательно)

После успешной записи файла запроса **в том же сообщении (или сразу следующим блоком)** выведи пользователю:

1. **Полный `id`** — строка целиком, **без сокращений** (например `9680b0ff-8174-4a81-87f7-c31af1e1fe6a`), чтобы её можно было скопировать и позже сослаться в чате («покажи результат задачи …»).
2. **Путь к файлу запроса** от корня репозитория: обычно `inbox/{id}.json`; для черновика — фактический путь вида `docs/drafts/inbox-{id}.json`.
3. **Где будет результат после воркера** (напоминание одной строкой): машинный ответ — `outbox/res_{id}.json`; при конверте — также каталог `outbox/{id}/` (см. `docs/SYSTEM_INSTRUCTIONS.md` про паттерн конверта).

Это не исполнение очереди и не объявление задачи выполненной — только явная метка для оператора и для последующих запросов в чате.

## Не делать

- **Не совмещать с воркером:** не создавать и не изменять файлы в `processing/`, `outbox/`, `archive/` и не объявлять задачу выполненной только потому, что создан `inbox/*.json`.
- Не использовать «копировать и удалить» вместо **атомарного `rename`** при **исполнении роли принимающего агента** (это не этот скилл; см. `docs/ATOMIC_CAPTURE.md`). Этот скилл **только создаёт** входящее сообщение.
- Не перезаписывать существующие `inbox/*.json`.
- Не подставлять **случайный** `parent_id`: только явный от пользователя или однозначный из контекста (и с проверкой **`outbox/res_{parent_id}.*`** при возможности).
- Не подставлять чужие `id` и не смешивать несколько запросов в одном файле.
