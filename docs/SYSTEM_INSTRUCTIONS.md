# Системные инструкции для агентов A2A (обязательны к соблюдению)

Документ задаёт **нормативную модель поведения** любого автономного агента, работающего в репозитории с файловой шиной. Нарушение инструкций считается ошибкой выполнения.

---

## 0. Термины и объектная модель

- **Запрос (Request)** — файл JSON-RPC 2.0, размещаемый в `inbox/` или перенесённый в `processing/`.
- **Ответ (Response)** — **логический** JSON-RPC 2.0 объект по разделу 2.2; **обязательный** машиночитаемый носитель — файл **`outbox/res_{id}.{ext}`** (раздел 2.3.2).
- **Дополнительные артефакты** — любые конечные файлы выдачи **помимо** машинного ответа §2.2 (отчёты в PDF/MD, таблицы, изображения, логи и т.д.). Временные и промежуточные файлы **не** считаются артефактами публикации.
- **Конверт** — **опциональный** каталог `outbox/{id_запроса}/`, вводится **только** при наличии **хотя бы одного** дополнительного артефакта; состав — §2.3.3.
- **Идентификатор запроса** — значение поля `"id"` в теле запроса; имя файла запроса **должно** быть `{id}.json` (без пробелов; допустимы UUID в канонической строковой форме).
- **Состояния жизненного цикла запроса:**
  - **PENDING** — файл находится только в `inbox/`.
  - **LOCKED** — файл находится только в `processing/` (задача принята конкретным агентом).
  - **COMPLETED** (логическая фаза) — для исходного запроса: выдача опубликована в `outbox/` по паттерну Конверта; исходный запрос перемещён в `archive/`.

Инвариант: В каждый момент времени для данного `{id}` существует **не более одного** файла запроса в совокупности каталогов `inbox/` и `processing/`.

---

## 1. Предварительные условия перед циклом

1. Агент **обязан** определить корень репозитория Git (далее `$REPO`).
2. Агент **обязан** прочитать `config/a2a-environment.json` и использовать пути каталогов относительно `$REPO`.
3. Агент **обязан** зарегистрировать или обновить свой манифест в `$REPO/.well-known/agents/{agent_id}.json`, где `{agent_id}` — уникальный идентификатор процесса агента; поле `inbox_path` **должно** совпадать с конфигурацией (`inbox/`).
4. Агент **не имеет права** модифицировать файлы другого агента в `processing/`, если имя файла уже существует и не совпадает с задачей, захваченной данным агентом.

---

## 2. Спецификация формата сообщений

### 2.1. Запрос (файл в `inbox/` или `processing/`)

Поля верхнего уровня (обязательность по строке «Требование»):

| Поле | Тип | Требование |
|------|-----|------------|
| `jsonrpc` | string | Ровно `"2.0"`. |
| `method` | string | Непустая строка. |
| `params` | object | Объект (может быть пустым `{}`). |
| `id` | string | Уникальный идентификатор; совпадает с именем файла без `.json`. Для **доработки** рекомендуется **новый UUID v4**; **не** кодировать номер итерации в `id` (связь с предком — **`parent_id`**, раздел 7). |
| `parent_id` | string \| **отсутствует** | **Опционально** для корня цепочки. Для **доработки (новая итерация)** — **обязательно**: строка, **строго** равная полю `id` **непосредственно предшествующей** итерации в дереве (см. раздел 7). |
| `sender_id` | string | Идентификатор отправителя. |
| `timestamp` | string | ISO-8601. |

Запрет: в одном файле **не допускать** смешение нескольких JSON-RPC объектов или постороннего текста до/после JSON.

### 2.2. Логическая модель ответа JSON-RPC 2.0

Машинное тело ответа **всегда** читается из **`outbox/res_{id}.json`** (или иного `ext` по §2.3.2). Структура объекта:

| Поле | Тип | Требование |
|------|-----|------------|
| `jsonrpc` | string | Ровно `"2.0"`. |
| `result` | null \| object | При успешной обработке — объект результата; при ошибке на уровне протокола A2A — допускается `null` только если заполнено `error`. |
| `error` | null \| object | При успехе — **явно** `null`. При ошибке — объект JSON-RPC ошибки `{ "code": integer, "message": string, "data"?: ... }`. |
| `id` | string | **Строго** равно `id` исходного запроса. |
| `processed_by` | string | **Ровно** `agent_id` обработавшего агента. |

Инвариант: при ошибке исполнения метода использовать `error` и задавать `result: null`. При успехе — `error: null`.

**Физическое размещение** этого объекта задаётся **только** разделом 2.3.

### 2.3. Паттерн Конверта (физическая выдача в `outbox/`)

Перед публикацией агент **обязан**:

1. Сформировать объект ответа §2.2.
2. Определить, есть ли **дополнительные артефакты** (§0), которые недопустимо или нежелательно целиком помещать в `result`.

**Инвариант:** файл **`outbox/res_{id}.{ext}`** с телом §2.2 обязателен **всегда** (§2.3.2). Каталог **`outbox/{id}/`** (конверт) создаётся **только** если дополнительные артефакты есть; без них конверт **не** создаётся (в том числе не создают «пустой» `metadata.json` с пустым `files`).

#### 2.3.1. Порядок в корне `outbox/`

В корне `outbox/` **не допускается** мусор и произвольные имена. Допустимы **только**:

- файлы вида **`res_{id}.{ext}`** — по одному на каждую завершённую задачу, с тем же `{id}`, что и запрос;
- **опционально** каталоги **`{id}/`** — конверты для тех же `id`, для которых есть дополнительные артефакты.

Сборка незавершённого конверта и файлы `*.tmp` **не** ведутся «в лоб» в корне `outbox/` (см. §2.3.4).

#### 2.3.2. Обязательный файл результата

Машинный ответ §2.2 **всегда** публикуется по пути:

```text
$REPO/outbox/res_{id}.{ext}
```

где `{id}` — идентификатор запроса, `{ext}` — по типу содержимого (для JSON-RPC из §2.2 используйте **`json`**).

Имя файла: **`res_{id}.{ext}`** (литерал префикса `res_`).

Запись: временный файл на том же томе, например `$REPO/outbox/res_{id}.json.tmp`, затем **одно** атомарное `rename` → `res_{id}.json`.

#### 2.3.3. Опциональный каталог-конверт

Если есть **хотя бы один** дополнительный артефакт, агент **должен** дополнительно создать каталог:

```text
$REPO/outbox/{id}/
```

Внутри **обязаны** находиться:

1. Все дополнительные артефакты (не дублирующие содержимое `res_{id}.json`, если это уже полный машинный ответ).
2. Файл **`metadata.json`** — «опись вложения» со **строго** следующей структурой верхнего уровня:

```json
{
  "jsonrpc": "2.0",
  "id": "{id_запроса}",
  "processed_by": "{agent_id}",
  "timestamp": "{ISO8601}",
  "summary": "Краткое описание выполненной работы",
  "files": [
    {
      "name": "filename.ext",
      "type": "тип_файла",
      "description": "Описание назначения файла"
    }
  ]
}
```

где `id`, `processed_by`, `timestamp` согласованы с задачей и исполнителем, массив **`files`** описывает **только файлы внутри этого каталога**, кроме самого **`metadata.json`** (служебный файл в список **не** включается).

Запрещено класть в конверт **дубликат** машинного JSON-RPC ответа под именем `response.json` или иным — канонический объект §2.2 находится **только** в **`outbox/res_{id}.json`**.

#### 2.3.4. Атомарность и порядок публикации

- **Только `res_{id}.json`:** одна атомарная публикация по §2.3.2.
- **С конвертом:** собрать `$REPO/outbox/{id}/` во временном каталоге **вне** `outbox/`, рекомендуется `$REPO/.a2a-outbox-staging/{id}/` (в `.gitignore`). После готовности всех вложений и `metadata.json` выполнить **одно** атомарное `rename` стейджинга → `$REPO/outbox/{id}/`. Затем опубликовать **`res_{id}.json`** (или наоборот: сначала `res_*.tmp` → `res_`, потом конверт — главное, чтобы к моменту **переноса запроса в** `archive/` и **`git commit`** присутствовали **и** итоговый `res_{id}.{ext}`, **и** при необходимости полный каталог конверта).

Вся выдача **должна** быть завершена **до** `git commit` по данной задаче.

#### 2.3.5. Обнаружение ответа подписчиком

1. **Всегда:** прочитать `$REPO/outbox/res_{id}.json` (или иной заявленный `ext`) и распарсить как объект §2.2.
2. **Если существует** `$REPO/outbox/{id}/`: прочитать `metadata.json` для сводки и списка дополнительных файлов; сами файлы — по путям относительно каталога конверта.

## 3. Рабочий цикл (обязательный алгоритм)

Активации агента идут **проходами**. **Один проход** — неделимая сессия шагов ниже от синхронизации с репозиторием (**pull**/`fetch`, если применимо) до завершения прохода. Между проходами **не предполагается** непрерывный внутренний **poll** или sleep процесса агента; следующий проход инициирует **внешний** оркестратор (**cron**, **systemd.timer**, CI, оператор-человек или иной supervisor), если оператор явно не настроил иное.

### Инвариант: не более одной успешной обработки за проход

За один проход запрещено успешно завершить (**шаг D**) **более одной** задачи (`request_id`). После успешного **D** проход **немедленно** завершается; любой следующий файл в `inbox/` может участвовать только в **следующем** проходе, инициированном **снаружи**.

Внутри одного прохода в шаге A допускается **перебор** нескольких кандидатов **только** для отказа («не принимаю задачу» без захвата) или попытки захватить задачу, пока **ещё не** достигнуто успешное **D**; как только успешное **D** выполнено, перебор **запрещён**.

### Шаг A — Выбор задачи

1. Составить упорядоченный список имён файлов `*.json` в `inbox/` (детерминировать сортировку: по имени файла лексикографически **или** по `(timestamp из JSON, затем id)` — **выбрать одну стратегию и зафиксировать её в журнале агента навсегда**).
2. В порядке списка, для каждого кандидата: при необходимости **прочитать** запрос без захвата и решить, принимает ли данный процесс задачу к исполнению (семантика `method`/`params`, возможности процесса, политики). Если задача **не** принимается — перейти к следующему кандидату **без** `rename` из `inbox/`.
3. Если задача принята — выполнить **атомарный захват** согласно `docs/ATOMIC_CAPTURE.md` (`rename` **`inbox/{id}.json` → `processing/{id}.json`**). При исходе «проигрыш гонки» источника исчез — следующий кандидат того же списка; при коллизии `processing/` — эскалация человеку.
4. Если захват успешен — перейти к шагу B с `request_id`, совпадающим с именем файла без `.json` и полем `id`.
5. Если список кандидатов исчерпан без успешной цепочки до шага **D** — завершить проход.
6. **После любого успешного шага D** не возвращаться к пунктам 2–5 **в том же** проходе.

### Шаг B — Обработка

1. Прочитать `$REPO/processing/{request_id}.json`, распарсить JSON.
2. Валидировать обязательные поля запроса (раздел 2.1). Если задано **`parent_id`** — проверить соответствие политике итераций (**раздел 7**): идентификатор текущей задачи **`id`** **не должен** совпадать с `parent_id`; при нарушении — эскалация оператору.
3. **Преемственность (если в запросе есть `parent_id`):** до исполнения `method` **обязательно** собрать **базовый контекст** предшествующей итерации (**раздел 7.2**). Если однозначно восстановить контекст нельзя (нет `outbox/res_{parent_id}.*` и нет согласованных артефактов в `outbox/{parent_id}/`, а восстановление из `archive/` не снимает неопределённость) — **остановить** автономную обработку и **запросить решение у человека** с перечислением проверенных путей.
4. Выполнить семантику `method` и `params` в рамках объявленных в манифесте `capabilities`, трактуя собранный базовый контекст как **уже принятую** работу прошлой итерации (**раздел 7.3**).
5. Подготовить объект ответа по разделу 2.2.
6. **Перед записью в `outbox/`:** спланировать публикацию по §2.3: **всегда** — `res_{request_id}.{ext}` с объектом §2.2; **при наличии дополнительных артефактов** — также каталог `outbox/{request_id}/` с `metadata.json` и вложениями. Имя **`request_id`** — **только** из текущего запроса; выдача предыдущей итерации **не** перезаписывается (**раздел 7**).

### Шаг C — Публикация выдачи и архивирование запроса

1. Опубликовать выдачу в `outbox/` согласно **§2.3** (атомарно, без мусора в корне `outbox/`).
2. Если публикация завершилась неуспешно из-за **конкуренции**, **EBUSY**, **EIO** или иной ошибки ввода-вывода, агент **обязан** остановить автономную обработку этой задачи и **явно сообщить оператору-человеку** полный текст ошибки и пути файлов (**запрещены** бесконечные молчаливые повторы).
3. **После** полной успешной публикации выдачи в `outbox/` атомарно переместить `$REPO/processing/{request_id}.json` → `$REPO/archive/{request_id}.json`.

Инвариант: в `archive/` запрос **не** перемещают, пока не опубликован **обязательный** `res_{request_id}.{ext}`; если по задаче предусмотрен конверт — пока не готов **весь** каталог `outbox/{request_id}/` (включая `metadata.json` и все перечисленные в нём файлы).

### Шаг D — Фиксация версии состояния (Git)

После успешного завершения шага C для данной задачи:

1. Выполнить в `$REPO`: `git add -A` либо **точечно** добавить изменённые пути: **`outbox/res_{request_id}.*`**, при наличии — каталог **`outbox/{request_id}/`**, перемещённый архивный файл, любые согласованные записи в `.gitignore`/документации, **не** добавляя незавершённые стейджинг-каталоги.
2. Выполнить `git commit` с сообщением, равным **конкатенации** литералов: префикс `Processed ` (включая завершающий пробел) и строкового значения `{id}` исходного запроса; пример сообщения для `id=abc-123`: `Processed abc-123`.
3. Выполнить `git push` в настроенный `remote` (**если** push невозможен из-за конфликта с удалённой веткой — раздел 5).

## 4. Ошибки, отказы и взаимодействие с человеком

1. Если на любом шаге чтение / запись / перемещение файла вернуло ошибку блокировки, отказ диска, или файл исчез неожиданно — агент **обязан**:
   - зафиксировать трассировку (путь, код/текст ошибки ОС, шаг алгоритма);
   - **обратиться к человеку** с явным вопросом и предложением действий;
   - **не** утверждать успешность обработки.
2. Если бизнес-логика метода завершилась ошибкой домена — агент **обязан** сформировать объект §2.2 с заполненным `error`, `processed_by`, `result: null`, **всегда** опубликовать его в **`outbox/res_{id}.json`**, при наличии дополнительных файлов по ошибке — добавить конверт по §2.3.3 (в `summary`/`description` отразить суть ошибки), затем выполнить шаг C и шаг D.

---

## 5. Конкурентный доступ через Git

1. Перед началом работы над очередью агент **рекомендуется** выполнять `git pull` (или политику, утверждённую оператором).
2. После локального `commit`, при отказе `push` (**non-fast-forward**) агент **обязан**:
   - **не** выполнять `git push --force` без прямого письменного указания оператора;
   - выполнить `git pull` (или утверждённое слияние);
   - при **конфликте содержимого** в любой из операционных очередей `inbox/` / `processing/` / `outbox/` — **немедленно остановиться** и **запросить решение у человека** (**запрещено** автономное разрешение конфликтов в этих путях без явной процедуры оператора).

---

## 6. Запреты (инварианты безопасности)

1. Запрещено удалять чужие запросы из `processing/`.
2. Запрещено перезаписывать уже существующую финальную выдачу для того же `id` (**`res_{id}.*`** и при наличии каталог **`outbox/{id}/`**) без решения оператора.
3. Запрещено записывать результат в `outbox/` до успешного атомарного захвата (файл **должен** присутствовать как `$REPO/processing/{request_id}.json`).
4. Запрещено хранить секреты и ключи в файлах открытой шины; для секретов **должны** использоваться только одобренные оператором секрет-хранилища вне этих каталогов.
5. **Итерации (раздел 7):** запрещено изменять или «чинить» **чужую** финальную выдачу и архивные запросы предков: **`outbox/res_{parent_id}.*`**, **`outbox/{parent_id}/`** и **`archive/{parent_id}.json`** остаются **неизменными** историческими артефактами; доработка оформляется **новым** `id` и **новой** выдачей (**раздел 7**).

## 7. Iteration Policy (итерации и преемственность)

Репозиторий A2A переводит **линейный** конвейер «один `id` — одна выдача» в **итеративную** модель: каждая доработка — **новая** вершина в истории Git, связанная с предшественником через **`parent_id`**, без перезаписи прошлых артефактов.

### 7.1. Дерево итераций и именование

- Каждая обработанная задача образует **узел** с ключом **`id`** (имя файла запроса без `.json`).
- Поле **`parent_id`** задаёт **ровно одного** непосредственного предка (**ребро** `parent_id → id` в дереве). Корневая задача **`parent_id` не содержит** (или по согласованию с оператором использует явный маркер в `params`; в протоколе по умолчанию — **отсутствие поля**).
- **Соглашение по `id`:** для **каждой** новой итерации генерируйте **новый** уникальный идентификатор (рекомендуется **UUID v4** в канонической строковой форме). **Связь с предком** задаётся **только** полем **`parent_id`**, без суффиксов `v2`/`v3` в **`id`**.
- Несколько дочерних задач с **одним** `parent_id` допустимы (ветвление); каждая дочерняя — **своим** уникальным **`id`** и **своим** `outbox/res_{id}.*` (и при необходимости **своим** конвертом `outbox/{id}/`).

### 7.2. Обязанность поиска базового контекста при `parent_id`

Если в запросе присутствует **`parent_id`**, исполнитель **до** семантики `method` **обязан**:

1. Проверить наличие машинного ответа предка: **`$REPO/outbox/res_{parent_id}.*`** (канонически **`res_{parent_id}.json`**).
2. Если существует **`$REPO/outbox/{parent_id}/`**, прочитать **`metadata.json`** и загрузить перечисленные файлы как часть контекста (аналогично **§2.3.5**).
3. Прочитать **`$REPO/archive/{parent_id}.json`**, чтобы восстановить **исходную постановку** предка (`method`, `params`, комментарии), если это нужно для непротиворечивого продолжения.
4. **Не** считать найденным контекстом «черновик» предка в `inbox/` или незавершённый захват в `processing/` — при отсутствии зафиксированной выдачи в **`outbox/`** по `parent_id` доработка **не** продолжается автономно (**шаг B п.3**).

Найденные данные образуют **базовый контекст**: всё, что уже было сделано и опубликовано для `parent_id`, трактуется как **несбрасываемая** основа; новое требование из текущего запроса **накладывается** сверху.

### 7.3. Склеивание контекста: старое + новое требование

Чтобы **не потерять** работу предшествующей итерации, исполнитель **обязан**:

1. **Структурировать вход на три слоя:**
   - **A — базовый контекст:** содержимое **`res_{parent_id}.*`** (структура `result` / текст ошибки / метаданные), плюс файлы из **`outbox/{parent_id}/`**, плюс при необходимости тело **`archive/{parent_id}.json`** (исходная формулировка задачи предка).
   - **B — текущая постановка:** поля **`method`**, **`params`** **этого** запроса (включая человекочитаемые поля вроде **`params.comment`**, **`params.change_request`**, уточнения ограничений).
   - **C — разрешение конфликтов:** если новое требование **явно** отменяет часть старого — в ответе **`result`** (или в `summary` конверта) **кратко** указать, что сохранено из A, что заменено по B, чтобы **audit trail** оставался читаемым в одном коммите `Processed <новый_id>`.
2. **Не дублировать слепо** большие артефакты предка в новый конверт без необходимости: допустимо ссылаться на пути предка в тексте ответа; **если** политика задачи требует самодостаточного пакета — копировать или пересобрать содержимое **внутри нового** `outbox/{id}/`, **не** изменяя файлы предка.
3. Публиковать результат **только** под **`id` текущего** запроса: **`outbox/res_{id}.*`** и при необходимости **новый** каталог **`outbox/{id}/`** (никогда не подменять выдачу предка, публикуя доработку в путях **`{parent_id}`**).

### 7.4. Запрет на перезапись истории и Git-прослеживаемость

- **Запрещено** править уже опубликованные **`outbox/res_{parent_id}.*`**, каталог **`outbox/{parent_id}/`** и заархивированный **`archive/{parent_id}.json`**, чтобы «подтянуть» доработку: исправление — **только** новым узлом (**§6 п.5**).
- Каждая успешная итерация по-прежнему завершается **шагом D** с сообщением **`Processed {id}`**; цепочка коммитов отражает рост дерева итераций, а **`git revert`** до выбранного коммита восстанавливает наблюдаемое состояние выдачи на тот момент **без** уничтожения истории на удалённой ветке обычным push (см. **`docs/VERSIONING.md`**).

### 7.5. Постановка доработки человеком

Оператор создаёт **новый** файл **`inbox/{новый_id}.json`**, где:

- **`id`** = новый уникальный идентификатор (рекомендуется **UUID v4**);
- **`parent_id`** = **`id`** итерации, от которой отталкиваемся (например `req-98765` или UUID предка);
- в **`params`** — явное описание доработки (например **`comment`**, **`delta`**, **`acceptance_notes`**).

Пример тела запроса см. **`docs/examples/request-iteration-with-parent-id.json`**.
