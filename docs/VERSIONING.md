# Версионирование состояния A2A через Git

## 1. Что называется «снимком состояния»

**Снимок** — это дерево объектов Git (деревья и связанные с ними blobs), которое полностью задаёт множество отслеживаемых файлов и их содержания на данном узле истории. Коммит **неизменяемо связывает** этот набор с предыдущим родительским объектом через хеш объекта.

В контексте A2A к снимку относятся:

- множества JSON-RPC файлов всех активных состояний `inbox/`, `processing/`, `outbox/`, `archive/`;
- манифесты `.well-known/agents/*.json`;
- конфигурация `config/a2a-environment.json`;
- любые добавленные в репозиторий связанные артефакты.

## 2. Как коммиты выражают эволюцию очередей

При соблюдении `docs/SYSTEM_INSTRUCTIONS.md` каждая финализируемая задача приводится к патчу:

- появился `outbox/{id}.json`;
- файл запроса исчез из `processing/{id}.json`;
- файл запроса появился как `archive/{id}.json`.

Далее `git commit` упаковывает эти переходы в **один неизменяемый объект коммита**, чей родитель — предыдущий коммит ветви по политике линейного или нелинейного продвижения.

Сообщения коммита шаблоном `Processed <id>` задают простой связующий указатель между бизнес-идентификатором задачи и хэшем ревизии.

## 3. Глобальная согласованность между агентами

1. После успешного `push` наблюдаемые клонированием узлы воспроизводят дерево операции `git fetch` + fast-forward-merge (или утверждённый merge).
2. Текущее **имя ветви** («где HEAD») + хэш коммита = идентификатор наблюдаемого снимка.
3. Если два агента параллельно продвинули одну линию вперёд расходящимися коммитами, история имеет два родителя и требуется явное решение оператора через merge или rebase; **автономное** слияние JSON в очередевых директориях не допускается.

## 4. Простая метрика времени воспроизведения

Ответ на вопрос «когда система впервые стала наблюдаемой в состоянии S?» находится через:

```
git log --reverse --pretty=format:'%h %ci %s'
```

со сравнением деревьев `git diff <parent>..<child> -- inbox processing outbox archive`.

Коммиты фиксируют **глобально опубликованные** состояния; локальная фаза «задача уже в processing, результат ещё не записан» **не является** членом наблюдаемой истории до очередного `commit`.

### Ключевая абстрактная формула

$$\text{A2A состояние } S_t^{\text{(global)}} = \text{Trees}(C_{t}^{\text{pushed}}),$$ где \(C_{t}^{\text{pushed}}\) — коммит успешной операции синхронизации с remote после обработки очередной задачи.

## 5. Эксплуатационные рекомендации для чистых снимков

1. **Commit ровно на одну финализируемую задачу**, чтобы сообщение `Processed <id>` биективно сопоставлялось патчам.
2. **Не использовать** `push --force` на основную ветку.
3. **Тегировать** зафиксированную политику инструментов человеком например аннотирующим им тегам.

---

Краткая формулировка для операторов: **каждый успешный `git push` публикует полный воспроизводимый снимок набора JSON-RPC файлов очередей** в данной ревизии дереве репозитория любой наблюдатель получает точно такое же дерево объектов при клонировании или fetch.
