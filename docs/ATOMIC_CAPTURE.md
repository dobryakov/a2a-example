# Атомарный захват задачи (исключение коллизий между агентами)

Цель — гарантировать, что **не более одного** автономного агента обрабатывает конкретный `{request_id}` в данный интервал времени при использовании файловой очереди и Git как журнала версий.

---

## 1. Обозначения

- `$REPO` — абсолютный корень клона этого репозитория Git.
- `SOURCE = join($REPO, "inbox", request_id + ".json")`.
- `TARGET = join($REPO, "processing", request_id + ".json")`.

---

## 2. Операция «атомарного захвата»

### 2.1. Препозиционные условия (обязательны)

До вызова `rename` необходимо (в любом допустимом порядке, но завершить до `rename`):

1. Вычислить `SOURCE` для выбранного кандидата `request_id`; имя файла и значение `"id"` в теле JSON **должны** совпадать.
2. Если существует `TARGET`, переход считается **блокирующим конфликтом**: автономно перезаписывать `TARGET` запрещено (см. §3.2).

### 2.2. Примитив

**Единственно допустимый** переход состояния PENDING → LOCKED:

```
rename(SOURCE, TARGET)
```

Под `rename` понимается **одна** операция системного вызова (или её эквивалент платформы) перемещения объекта имени без пользовательской «композиции» из копирования и удаления.

Копировать содержимое и удалять источник **две отдельные операции** как основную стратегию нельзя: это создаёт окно двойного успеха двух процессов.

### 2.3. Постусловия и классы исходов

Обозначим результат `rename` как `errno`/`status`:

| Исход | Признак | Действия агента |
|-------|---------|-----------------|
| **CLAIMED** | Успешный результат и после вызова `exists(TARGET)`, `not exists(SOURCE)` | Исполнение метода; состояние **LOCKED**. |
| **LOST_RACE** | Отсутствие `SOURCE` до вызова **или** код `ENOENT`/эквивалент («нет такого источника») | Считать, что другой победитель взял файл; продолжить цикл очереди. |
| **CONFLICT_EXISTS** | Код `EEXIST`/эквивалент о целевом имени или заранее обнаруженный существующий `TARGET` | Не изменять файл в `processing/`; **немедленно эскалация оператору-человеку** по правилам `docs/SYSTEM_INSTRUCTIONS.md`. |
| **IO_FAILURE** | Любой иной код ошибки транспортного уровня | Остановиться; сообщить оператору сырые коды ошибок без циклических ретраев. |

Гонка между проверкой `exists(TARGET)` и `rename` допускает редкий `EEXIST`; классифицировать её как **CONFLICT_EXISTS**.

---

## 3. Свойства атомарности (POSIX, один том)

На локальной POSIX-ФС переименование внутри **одной** точки монтирования **атомарно**:

- Если два процесса одновременно стремятся перенести **идентичный** `SOURCE`, ровно **один** успех; второй наблюдает исчезновение источника (**LOST_RACE**).
- Различные файлы перемещать параллельно безопасно.

Git здесь только фиксирует **историю после** набора локальных мутаций: он **не является** блокировкой гонки `rename`; защита исключительно файловая.

---

## 4. Интеракция с синхронизируемыми/сетевыми ФС

NFS версий с ограниченной семантикой, синхронизаторы («облако» типа Dropbox) между узлами, или смешённые тома ломают гарантии §3 Если нельзя доказать **атомарный rename на одном томе**, нужна **политика развёртывания** (общий блокирующий брокер, выделенный узел-хранитель очередей) одобренная оператором.

Docker bind mount между контейнером и хостом **не отменяет** атомарность `rename`, если том один и локальный.

---

## 5. Псевдокод для машинной интерпретации

```
AtomicClaim(request_id):
  SOURCE = path(REPO/inbox/request_id.json)
  TARGET = path(REPO/processing/request_id.json)

  if EXISTS(TARGET): return CONFLICT_EXISTS

  if not EXISTS(SOURCE): return LOST_RACE

  err = POSIX_RENAME(SOURCE, TARGET)
  if err == 0:
    CHECK EXISTS(TARGET) and NOT EXISTS(SOURCE)
    return CLAIMED

  if err implies missing SOURCE after race:
    return LOST_RACE
  if err implies TARGET collision:
    return CONFLICT_EXISTS

  return IO_FAILURE
```
